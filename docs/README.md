# Sistema de Recomendaciones Inteligentes - ETERNA DX

## Visi√≥n general

**ETERNA DX Rules Engine** es un sistema completo de recomendaciones personalizadas para pacientes basado en datos de actividad f√≠sica, sue√±o y bienestar. El sistema eval√∫a reglas configurables contra m√©tricas derivadas de datos biom√©tricos para generar recomendaciones contextuales y personalizadas.

### Prop√≥sito y alcance

- **Prop√≥sito**: Generar recomendaciones inteligentes de salud y bienestar basadas en patrones de actividad, sue√±o y m√©tricas fisiol√≥gicas
- **Dominios clave**:

  - Procesamiento de datos biom√©tricos y derivaci√≥n de features
  - Sistema anti-repetici√≥n inteligente
  - Interfaz administrativa web
  - API REST completa con auditor√≠a

### Tecnolog√≠as principales

- **Backend**: FastAPI (Python 3.9+), SQLAlchemy, Pydantic
- **Frontend**: Next.js 14, React 18, TypeScript, TailwindCSS  
- **Base de datos**: SQLite (desarrollo), escalable a PostgreSQL/MySQL
- **Procesamiento**: Pandas, NumPy para an√°lisis de series temporales
- **Infraestructura**: Uvicorn, PowerShell scripts

### Puntos de entrada

- **API REST**: `http://127.0.0.1:8000` - Endpoints para evaluaci√≥n, gesti√≥n de reglas y analytics
- **Admin UI**: `http://127.0.0.1:3000` - Interfaz web para gesti√≥n de reglas y visualizaci√≥n
- **Scripts CLI**: `scripts/10_load_and_merge.py` - Procesamiento de datos CSV
- **Simulaci√≥n**: Endpoint `/simulate` para testing de reglas contra fechas espec√≠ficas

### Riesgos t√©cnicos principales

‚ö†Ô∏è **Seguridad**: Autenticaci√≥n deshabilitada por defecto (`auth_enabled: false`)
‚ö†Ô∏è **Rendimiento**: Procesamiento s√≠ncrono de features por usuario/fecha
‚ö†Ô∏è **Escalabilidad**: SQLite no apto para producci√≥n con m√∫ltiples usuarios concurrentes

### Rendimiento

üöÄ **Hotspots de CPU/memoria**
- **`build_features()`**: O(N*W) - cachear por usuario/fecha si es posible
- **`evaluate_user()`**: O(R*C) - considerar lazy loading de reglas
- **Rolling windows**: Pandas `.rolling()` - usar ventanas fijas vs expandidas
- **CSV loading**: `pandas.read_csv()` - considerar chunking para archivos >100MB

üíæ **Optimizaciones de I/O**
- **Database**: √çndices en `user_id`, `date`, `rule_id` para queries frecuentes  
- **CSV caching**: Cargar `daily_processed.csv` una vez por proceso
- **Batch processing**: Evaluar m√∫ltiples usuarios/fechas en lote

üîÑ **Puntos para cach√©**
- **Features por usuario**: Cache TTL=1h para datos del d√≠a actual
- **Reglas activas**: Cache invalidado solo en cambios de reglas
- **Variables metadata**: Cache permanente (raramente cambia)

üìä **Paginaci√≥n requerida**

- **`/rules/{id}/stats`**: Agregaciones sin ventana temporal
- **Admin UI**: Listas de reglas sin paginaci√≥n client-side

### Concurrencia

üîí **Secciones cr√≠ticas**

- **CSV processing**: No thread-safe - usar locks para escritura concurrente

## Glosario t√©cnico

- **ACWR**: Acute:Chronic Workload Ratio - Ratio entre carga de trabajo aguda (√∫ltimos 7 d√≠as) y cr√≥nica (√∫ltimos 28 d√≠as)
- **Anti-repetici√≥n**: Mecanismo que evita mostrar la misma variante de mensaje al mismo usuario en N d√≠as
- **Audit**: Registro de cada evaluaci√≥n de regla (disparada o no) con contexto completo
- **DSL**: Domain Specific Language - Lenguaje espec√≠fico para definir condiciones de reglas en JSON
- **Feature**: M√©trica derivada calculada a partir de datos raw (ej: rolling means, z-scores)
- **TRIMP**: Training Impulse - M√©trica de carga de entrenamiento basada en duraci√≥n e intensidad
- **Cooldown**: Per√≠odo m√≠nimo entre disparos de la misma regla para el mismo usuario
- **Tenant**: Entidad organizacional para multi-tenancy (ej: hospital, cl√≠nica)
- **Rolling window**: Ventana deslizante para c√°lculos estad√≠sticos (ej: media de 7 d√≠as)
- **Weighted random**: Selecci√≥n aleatoria donde cada opci√≥n tiene probabilidad proporcional a su peso
